**Fix 3 — Wire MCP push notifications (MCP-03):**

The challenge: `notifyJobStatusChanged` needs an `McpServer` instance, but the Inngest handler in `events.ts` runs in the engine package which doesn't have a reference to the MCP server. The cleanest approach is a callback pattern:

1. In `packages/engine/src/asset/events.ts`, extend the `AssetDeps` interface to include an optional notification callback:
```typescript
interface AssetDeps {
  db: DbClient;
  logger: Logger;
  executor: AssetExecutor;
  artifactsRoot: string;
  onJobStatusChanged?: (jobId: string) => void;  // NEW
}
```

2. In `generateAssetHandler`, call `onJobStatusChanged` after each successful state transition where an `appendAssetEvent` is called. Add calls after:
   - The `asset_job_active` event append (after line ~145): `getAssetDepsOrThrow().onJobStatusChanged?.(jobId);`
   - The `asset_job_completed` event append (after line ~244): `getAssetDepsOrThrow().onJobStatusChanged?.(jobId);`
   - The `asset_job_failed` event appends (lines ~152, ~164, ~251): `getAssetDepsOrThrow().onJobStatusChanged?.(jobId);`

3. In `packages/mcp/src/server.ts`, update `createMcpServer` to return both the server and a notification callback that callers can pass to `configureAssetDeps`:
   - Add an export: `export function createJobStatusNotifier(server: McpServer): (jobId: string) => void`
   - Implementation: `return (jobId) => notifyJobStatusChanged(server, jobId);`
   - Add import of `notifyJobStatusChanged` from `./resources/job-status.js`

This way, the MCP entry point (`index.ts`) can wire things together:
```typescript
const server = createMcpServer(deps);
const notifier = createJobStatusNotifier(server);
// Pass notifier to configureAssetDeps as onJobStatusChanged
```

Note: The MCP entry point wiring is informational — the actual `index.ts` change is optional since the MCP server and Inngest handler typically run in separate processes. The key deliverable is that the callback infrastructure exists and `notifyJobStatusChanged` is called in the handler when `onJobStatusChanged` is provided. In-process callers (like integration tests or a combined server) can wire it.

**Fix 4 — Robust template path resolution (packaging tech debt):**

In `packages/engine/src/asset/comfyui-adapter.ts`, replace `loadWorkflowTemplate()` with a resolution strategy that works both in monorepo and standalone:

```typescript
import { createRequire } from 'node:module';

function loadWorkflowTemplate(): string {
  // Strategy 1: Use Node module resolution via @get-cauldron/shared package
  // Works when shared is a proper node_modules dependency (standalone packaging)
  try {
    const require = createRequire(import.meta.url);
    const sharedPath = require.resolve('@get-cauldron/shared/package.json');
    const sharedRoot = dirname(sharedPath);
    const templatePath = resolve(sharedRoot, 'src/workflows/flux-dev.json');
    return readFileSync(templatePath, 'utf-8');
  } catch {
    // Strategy 2: Fallback to monorepo-relative path (dev environment)
    const __dirname = dirname(fileURLToPath(import.meta.url));
    const templatePath = resolve(__dirname, '../../../shared/src/workflows/flux-dev.json');
    return readFileSync(templatePath, 'utf-8');
  }
}
```

Remove the existing `__dirname` computation that is now inside the catch block. Keep the `fileURLToPath` import since it is used in the fallback.
