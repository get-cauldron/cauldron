    **Read the engine interview module thoroughly first** to discover exact function signatures, types, and how the InterviewFSM works. The tRPC router must call engine functions correctly.

    Create packages/web/src/trpc/routers/interview.ts with these procedures:

    1. **`getTranscript`** — Query: Given projectId, return the current interview state: transcript (messages), current scores (ambiguity per dimension), status (gathering/reviewing/approved/crystallized), current suggestions (MC options), and the active perspective.

    2. **`sendAnswer`** — Mutation: Given projectId and answer text, advance the interview FSM. Returns: next question, updated scores, new MC suggestions, perspective indicator, and whether clarity threshold is reached (overall ambiguity <= 0.2 per D-06).

    3. **`getSummary`** — Query: Given projectId, return the structured summary for review (when status is reviewing or approved).

    4. **`approveSummary`** — Mutation: Given projectId, approve the summary and trigger seed crystallization. Returns the new seed ID.

    5. **`rejectSummary`** — Mutation: Given projectId, reject the summary and return to interview gathering state.

    6. **`getHoldouts`** — Query: Given seedId, return holdout test scenarios for review (status, content per scenario).

    7. **`approveHoldout`** — Mutation: Given holdoutId, approve a specific holdout scenario.

    8. **`rejectHoldout`** — Mutation: Given holdoutId, reject a specific holdout scenario.

    9. **`sealHoldouts`** — Mutation: Given seedId, seal all approved holdout scenarios (triggers encryption).

    All procedures use `publicProcedure` with `z.object()` input validation. Each procedure calls the appropriate engine function via `@cauldron/engine` imports. If the engine does not expose a direct function, use Drizzle queries against the shared schema.

    **Update packages/web/src/trpc/router.ts** to merge `interviewRouter`:
    ```typescript

    ```
