    The function body implements the 8-state FSM per D-21:

    **Step 1 -- budget-check** (FSM pre-check):
    ```typescript
    try {
      await step.run('budget-check', async () => {
        const { db } = getEvolutionDeps();
        const limit = getEvolutionDeps().budgetLimitCents ?? 10000;
        await checkLineageBudget(db, seedId, limit);
      });
    } catch (err) {
      if (err instanceof BudgetExceededError) {
        await step.run('emit-budget-halt', async () => {
          await appendEvent(db, { projectId, seedId, type: 'evolution_halted', payload: { reason: 'budget_exceeded' } });
        });
        await step.sendEvent('trigger-holdout-unseal', { name: 'evolution_converged', data: { seedId, projectId, vaultId: '', codeSummary } });
        return { status: 'halted', reason: 'budget_exceeded' };
      }
      throw err;
    }
    ```

    **Step 2 -- load-seed** (FSM: evaluating):
    Fetch the seed from DB by seedId.

    **Step 3 -- evaluate-goal-attainment** (FSM: evaluating):
    Call `evaluateGoalAttainment({ gateway, seed, codeSummary, projectId, evolutionCycle: seed.generation, seedId })`.

    **Step 4 -- check-goal-met** (FSM: converged if score >= SUCCESS_THRESHOLD):
    If score >= 0.95:
    - Emit `evolution_goal_met` event via appendEvent
    - Send `evolution_converged` Inngest event (triggers holdout unseal via existing handleEvolutionConverged per Pitfall 3)
    - Return `{ status: 'converged', reason: 'goal_met' }`

    **Step 5 -- check-convergence** (FSM: scoring):
    Call `checkConvergence({ db, seedId, currentScore, currentGaps })`.
    If convergence halt:
    - Emit `evolution_halted` event via appendEvent with signal type
    - Send `evolution_converged` Inngest event for holdout unseal
    - Return `{ status: 'halted', signal: signal.type }`

    **Step 6 -- check-stagnation** (FSM: evolving / lateral_thinking):
    Fetch lineage, call `checkStagnation(lineage)`.
    If stagnant:
    - Emit `evolution_lateral_thinking` event
    - Call `runLateralThinking({ step, gateway, seed, gapAnalysis, projectId, seedId })`
    - If null (lateral thinking failed):
      - Emit `evolution_escalated` event (D-17)
      - Send `evolution_converged` for holdout unseal (D-19)
      - Return `{ status: 'halted', reason: 'escalated' }`
    - If non-null: use `mutateSeedFromProposal` to create the evolved seed directly from the lateral proposal:
      ```typescript
      const lateralSeed = await step.run('create-lateral-seed', () =>
        mutateSeedFromProposal({
          db, seed, proposal: lateralResult,
          projectId, seedId,
          lastScore: goalResult.overallScore,
          lastGapAnalysis: goalResult.gapAnalysis,
        })
      );
      ```
      Then dispatch decomposition with tier='full' (lateral thinking is always a full regen).

    **Step 7 -- generate-evolved-seed** (FSM: evolving, normal path):
    If NOT stagnant, call `mutateSeed({ db, gateway, seed, goalResult, projectId, seedId })`.
    Store the returned `newSeed` and the `tier` from `goalResult.tier`.

    **Step 8 -- dispatch-decomposition** (FSM: decomposing):
    Send Inngest event to trigger decomposition of the new evolved seed, passing tier and previousSeedId per D-08 (EVOL-04):
    ```typescript
    await step.sendEvent('trigger-decomposition', {
      name: 'bead.dispatch_requested',
      data: {
        seedId: newSeed.id,
        projectId,
        // Per D-08: ac_only evolution keeps completed beads from previous seed
        // Pass tier and previousSeedId so decomposition can skip already-completed beads
        tier: goalResult.tier, // 'full' or 'ac_only'
        previousSeedId: goalResult.tier === 'ac_only' ? seedId : undefined,
      },
    });
    ```
    When `tier='ac_only'`, the decomposition handler receives `previousSeedId` and MUST skip beads that were already completed for that seed. When `tier='full'`, no previousSeedId is passed, so decomposition runs all beads from scratch.

    Note: The decomposition handler (`beadDispatchHandler` in `packages/engine/src/decomposition/events.ts`) will need to respect the `previousSeedId` field to implement bead reuse. If it does not yet handle this field, it will safely ignore it (new beads dispatched for all). The actual bead-skipping logic is a downstream concern for the decomposition module -- this plan's responsibility is to PASS the data needed for the decision.

    Return `{ status: 'cycle_complete', nextSeedId: newSeed.id }`.
