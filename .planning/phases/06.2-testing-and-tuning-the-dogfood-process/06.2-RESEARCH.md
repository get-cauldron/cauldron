## Summary

Phase 06.2 is a debugging/tuning phase with two parallel tracks. The codebase is well-built — all 64 interview unit tests pass, the CLI toolchain is complete, and Docker services are running. The problems are behavioral, not structural: the LLM scorer consistently underscores concrete answers (14-24% despite specific inputs), the perspective activation logic keeps the interview stuck in early-phase questioning, and there is no mechanism to signal convergence pressure when one dimension is complete.

Track 1 (interview convergence) requires diagnosing and patching three distinct systems in `packages/engine/src/interview/`: the scorer system prompt calibration, the `selectActivePerspectives` thresholds, and a new convergence pressure signal. Track 2 (pre-generated seed pipeline) is a direct path to validating the downstream pipeline by inserting a hand-crafted seed row directly into the database, then running `cauldron seal`, `cauldron decompose`, and `cauldron execute` against it.

**Primary recommendation:** Run Track 2 first (pre-generated seed → downstream pipeline) — it is unblocked and directly validates the definition of done (at least one bead executing). Run Track 1 in parallel; both tracks must complete for the phase to be done.

## Convergence Bug Analysis

### Bug 1: Scorer Doesn't Reward Concrete Answers (CONFIRMED — HIGH confidence)

**Root cause identified in `scorer.ts`:**

The `SCORER_SYSTEM_PROMPT` is underdirected for calibration:
```
'Evaluate the interview transcript for clarity across dimensions. Score each dimension from 0 (completely unclear) to 1 (perfectly clear).'
```

This gives the LLM no anchor for what a "high" score means. The LLM is trained to be conservative about claiming 80%+ clarity. The dry-run showed oscillation at 14-24% even when the user provided concrete answers like "Unix developers, local-only, English-only" — inputs that should push goal clarity well above 50%.

**The fix target:** `SCORER_SYSTEM_PROMPT` in `scorer.ts` needs calibration guidance: what does 0.8 look like vs. 0.3? The system prompt must include examples or explicit anchoring criteria so the model understands that specific, testable answers warrant high scores.

**Secondary issue:** `buildScorerPrompt()` passes the full transcript but doesn't highlight the most recent answer. On turn 35+, the LLM context is dominated by early ambiguous turns. The scorer needs to weight recent answers more heavily, not treat all turns equally.

### Bug 2: Perspective Activation Stuck on Researcher (CONFIRMED — HIGH confidence)

**Root cause identified in `perspectives.ts`:**

`selectActivePerspectives()` has a turn count guard: when `turnCount === 0`, it always returns `['researcher', 'simplifier', 'breadth-keeper']` regardless of scores. After turn 0, it uses `overall` thresholds:

```typescript
if (overall < 0.4) return ['researcher', 'simplifier', 'breadth-keeper'];  // early
if (overall < 0.7) return ['architect', 'breadth-keeper', ...];            // mid
return ['seed-closer', 'architect'];                                        // late
```

Because the scorer underscores (bug 1), `overall` stays below 0.4 indefinitely. The perspective selector never advances past the early-phase set. `simplifier` and `seed-closer` never activate because we never reach `overall >= 0.4` or `overall >= 0.7`. **Bug 2 is downstream of Bug 1** — fix the scorer calibration first.

However, even with a fixed scorer, the thresholds may need tuning. If the scorer starts returning honest mid-range scores (0.4-0.6) for partial answers, the mid-phase perspectives should now activate. The key diagnostic: after fixing Bug 1, does `architect` and `breadth-keeper` start appearing?

### Bug 3: No Convergence Pressure (CONFIRMED — HIGH confidence)

**Root cause:** The FSM and perspective system have no mechanism to say "goal clarity is done, now focus on constraints." Each turn independently selects perspectives based on `overall` score. If goalClarity reaches 0.9 but constraintClarity is 0.2, the scorer might report `overall = 0.9*0.4 + 0.2*0.3 + 0.2*0.3 = 0.48` — mid-range — activating `architect` and `breadth-keeper`, which are generic. Nothing specifically targets the weakest dimension.

**The fix:** `selectActivePerspectives()` in mid-to-late turns must inspect dimension-level scores, not just `overall`. In mid turns, the "weakest dimension" logic already exists partially (it chooses `researcher` vs `simplifier` based on goalClarity). This needs to extend to all dimensions. In late turns, `seed-closer` should activate earlier if `successCriteriaClarity < 0.6` regardless of `overall`.

A convergence pressure signal could be: if a dimension score hasn't improved by >0.05 in the last 3 turns (computable from `ambiguityScoresHistory`), inject that dimension's specialist perspective explicitly.

## Track 2: Pre-Generated Seed Pipeline — Fix Map

### Seed Injection Strategy

The `crystallize.ts` command reads a seed draft JSON and calls `crystallizeSeed()`. The pre-generated seed can be injected by:
1. Writing a `.cauldron/review/seed-draft-<projectId>.json` file (matches `writeSeedDraft` path format) — then running `cauldron crystallize --project-id <uuid>` which will insert it into the DB.
2. Or directly using a one-off Drizzle script to INSERT.

Option 1 is cleaner — it uses the existing code path and validates the CLI crystallize command as a side effect.

### CLI Renamer Fixture Requirements

The fixture repo needs:
- A directory (e.g., `~/Projects/cli-renamer-fixture`) with at least `package.json`
- No `.planning/` directory (so context-bridge returns empty → greenfield mode for interview)
- No git commits OR an existing git repo with commits (brownfield — whichever the plan uses)
- A seed targeting: "Build a CLI tool that renames files by replacing a pattern in filename with a replacement string, recursively over a directory"

### Seed Content for CLI Renamer

The seed must satisfy the schema. Minimum content:
```json
{
  "goal": "Build a CLI file renamer tool that recursively renames files by replacing a pattern string in filenames with a replacement string within a target directory.",
  "constraints": [
    "TypeScript implementation",
    "No external LLM dependencies",
    "Unix/macOS target platform",
    "Dry-run mode must be supported (preview changes without applying)"
  ],
  "acceptanceCriteria": [
    "CLI accepts --dir, --find, --replace, --dry-run flags",
    "Recursively walks target directory",
    "Renames all files matching --find pattern in filename to --replace",
    "Dry-run mode lists changes without applying them",
    "Handles edge cases: no matches found, permission errors, name collisions"
  ],
  "ontologySchema": {
    "entities": [
      { "name": "File", "attributes": ["path", "name", "extension"], "relations": [] },
      { "name": "RenamePlan", "attributes": ["from", "to", "dry"], "relations": [{ "to": "File", "type": "targets" }] }

    "Correctness: renames exactly the files matched by the pattern, no others",
    "Safety: dry-run must never write to disk",
    "Clarity: output is human-readable"
  ],
  "exitConditions": {
    "allTestsPass": true,
    "dryRunVerified": true
  }
}
```

This produces a small, well-understood DAG (expected: 2-5 beads — setup, parser, walker, renamer, tests).

## Common Pitfalls

### Pitfall 1: Fixing the Wrong Layer First
**What goes wrong:** Patching `selectActivePerspectives` thresholds without fixing the scorer calibration. The perspective system is downstream of the scorer — if the scorer still returns 14-24%, the activation thresholds don't matter.
**Prevention:** Always run the scorer in isolation first. After a scorer prompt change, test: does the scorer return >0.5 for a transcript where the user gave concrete answers? If not, the scorer is still the bottleneck.

### Pitfall 2: Pre-Generated Seed Missing Required Project Row
**What goes wrong:** The `seeds` table has a FK on `project_id` referencing `projects`. Inserting a seed without a matching project row will fail with a FK violation.
**Prevention:** The seed injection script/command must first INSERT a row into `projects`. The `execute.ts` command also reads from `projects` via `deps.config.selfBuild`.

### Pitfall 3: Missing `settings` Column in projects Table
**What goes wrong:** Dry-run finding: "projects table missing `settings` column in dev DB." The workaround was raw SQL. This means migrations may not have run cleanly in the dev DB.
**Prevention:** Run `pnpm db:migrate` against the dev DB before any Track 2 work. Verify `\d projects` shows all columns including `settings`.

### Pitfall 4: Scorer Prompt Rewrite Breaks Existing Tests
**What goes wrong:** `scorer.test.ts` has tests that mock the gateway and check for `temperature: 0` and `stage: 'interview'`. If the rewrite changes the schema or call signature, these tests will fail.
**Prevention:** The scorer rewrite is prompt-only — `SCORER_SYSTEM_PROMPT` is a string constant. Schema (`greenfieldScoresSchema`) and gateway call signature do not change. Existing tests are unaffected. New tests should use an updated mock that returns scores appropriate to the new prompt.

### Pitfall 5: Crystallize Command Expects Specific Draft File Path
**What goes wrong:** `writeSeedDraft` writes to `.cauldron/review/seed-draft-<projectId>.json`. If the manually created seed draft is at a different path, `crystallize` won't find it.
**Prevention:** Either place the file at the exact path, or directly INSERT via a seed-injector script. The planner should specify which strategy.

### Pitfall 6: Inngest Server Must Be Running for `execute` Command
**What goes wrong:** `cauldron execute` sends events to Inngest. If the Inngest dev server is not running, events are dropped and beads never execute.
**Prevention:** Verify `docker compose ps` shows `cauldron-inngest-1` as `Up`. Current status: confirmed running at port 8288-8289.

### Pitfall 7: Scorer Temperature=0 vs. Prompt Changes
**What goes wrong:** The scorer uses `temperature: 0` for determinism. Changing the system prompt may produce different scores, but temperature=0 means the same input always produces the same output. During calibration testing, the tester must vary the transcript content to see score changes — not just re-run the same transcript expecting different output.
**Prevention:** Document: "After prompt change, run interview with fresh transcript from scratch — do not resume old session with cached scores."

## Open Questions

1. **Does fixing the scorer prompt alone unblock convergence?**
   - What we know: Scorer is the root cause; perspectives are downstream. A calibrated scorer should advance `overall` past 0.4 and 0.7 thresholds.
   - What's unclear: How conservative are frontier models at `temperature=0` for scoring? May still plateau below 0.8.
   - Recommendation: Fix scorer prompt first, run 1-2 full interviews, measure if score advances to 0.4+. If yes, perspective rotation should self-correct. If no, adjust thresholds.

2. **Does `interviewId` need to be non-null for a pre-generated seed?**
   - What we know: `seed.interviewId` is nullable (`references(() => interviews.id)` with no `notNull()`).
   - What's unclear: Whether downstream pipeline (decompose, execute) reads `interviewId` and fails on null.
   - Recommendation: Set `interviewId: null` in the pre-generated seed INSERT. If decompose fails, trace the error to the relevant field.

3. **Will the CLI renamer fixture need a `cauldron.config.ts`?**
   - What we know: Phase 6.1 added a fallback to Cauldron's own config when external projects lack `cauldron.config.ts`.
   - What's unclear: Whether `selfBuild: false` is set correctly in the fallback config.
   - Recommendation: Verify `captureEngineSnapshot()` is skipped when `selfBuild !== true` in execute.ts. Confirmed: `if (deps.config.selfBuild === true)` guards the self-build path.
