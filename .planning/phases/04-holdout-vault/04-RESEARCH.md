# Phase 4: Holdout Vault - Research

**Researched:** 2026-03-26
**Domain:** AES-256-GCM envelope encryption, cross-model test generation, process-level key isolation, Inngest event-driven unsealing
**Confidence:** HIGH

## Summary

Phase 4 builds the Holdout Vault: scenario test generation from crystallized seeds via a cross-model LLM call, AES-256-GCM envelope encryption (DEK/KEK), human review + two-step seal confirmation, and an unsealing protocol triggered by `evolution_converged` events. The codebase already has the `holdout_vault` table with all required encrypted-blob columns; the main work is writing the holdout module, extending the status enum, adding the key-isolation child-process test, and wiring the Inngest event handler.

The most important clarification from code review: Inngest `step.run()` does NOT provide per-step `process.env` isolation — all steps in a function share the same Node.js process. Key isolation is achieved architecturally by running the vault seal/unseal logic only inside a dedicated Inngest function that is deployed with `HOLDOUT_ENCRYPTION_KEY` in its env, while agent bead-worker Inngest functions are deployed (or spawned) without it. The integration test for isolation uses a child process to simulate the absent-key environment.

**Primary recommendation:** Build `packages/engine/src/holdout/` as the vault module. Use the existing gateway (`stage: 'holdout'`) for generation, `node:crypto` AES-256-GCM for encryption, extend `holdoutStatusEnum` via a manual migration following the established migration pattern from Phase 3, and register an Inngest event handler for `evolution_converged`.

### Pattern 1: AES-256-GCM Envelope Encryption
**What:** Random 256-bit DEK per holdout set; DEK encrypted with master KEK.
**When to use:** Sealing an approved scenarios array.
**Example:**
```typescript
// Source: node:crypto official docs — AES-256-GCM with GCM auth tag
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';

const ALGORITHM = 'aes-256-gcm';
const IV_LENGTH = 12;      // 96-bit IV is GCM standard
const KEY_LENGTH = 32;     // 256-bit

export function encryptWithDek(
  plaintext: string,
  dek: Buffer
): { ciphertext: string; iv: string; authTag: string } {
  const iv = randomBytes(IV_LENGTH);
  const cipher = createCipheriv(ALGORITHM, dek, iv);
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  return {
    ciphertext: encrypted.toString('base64'),
    iv: iv.toString('base64'),
    authTag: cipher.getAuthTag().toString('base64'),
  };
}

export function decryptWithDek(
  ciphertext: string,
  iv: string,
  authTag: string,
  dek: Buffer
): string {
  const decipher = createDecipheriv(ALGORITHM, dek, Buffer.from(iv, 'base64'));
  decipher.setAuthTag(Buffer.from(authTag, 'base64'));
  const decrypted = Buffer.concat([
    decipher.update(Buffer.from(ciphertext, 'base64')),
    decipher.final(),
  ]);
  return decrypted.toString('utf8');
}

export function encryptDek(dek: Buffer, kek: Buffer): { encryptedDek: string; dekIv: string; dekAuthTag: string } {
  // Same AES-256-GCM for the DEK itself; store separately
  const iv = randomBytes(IV_LENGTH);
  const cipher = createCipheriv(ALGORITHM, kek, iv);
  const enc = Buffer.concat([cipher.update(dek), cipher.final()]);
  return {
    encryptedDek: enc.toString('base64'),
    dekIv: iv.toString('base64'),
    dekAuthTag: cipher.getAuthTag().toString('base64'),
  };
}
```

### Pattern 3: Inngest Event Handler for evolution_converged
**What:** Register an Inngest function that listens for `evolution_converged` and unseals the vault.
**When to use:** Phase 4 builds the handler; Phase 7 triggers the event.
**Example:**
```typescript
// Source: Inngest v4 TypeScript SDK docs — createFunction + step.run
import { inngest } from './client.js';

export const handleEvolutionConverged = inngest.createFunction(
  { id: 'holdout-vault/unseal-on-convergence' },
  { event: 'evolution_converged' },
  async ({ event, step }) => {
    const { seedId, projectId } = event.data;

    // Key is available here because this function's deployment has HOLDOUT_ENCRYPTION_KEY
    const scenarios = await step.run('unseal-vault', async () => {
      return unsealVault(db, seedId);
    });

    const evalResult = await step.run('evaluate-holdouts', async () => {
      return evaluateHoldouts(gateway, scenarios, projectId, seedId);
    });

    if (!evalResult.passed) {
      await step.run('emit-failure-event', async () => {
        await appendEvent(db, {
          projectId,
          seedId,
          type: 'evolution_started',
          payload: { failureReport: evalResult.failureReport, triggeredBy: 'holdout_failure' },
        });
      });
    }
  }
);
```

### Anti-Patterns to Avoid
- **Storing the plaintext scenarios in DB before encryption:** The `holdout_vault` table has no plaintext column — pending review state is held in application memory or a separate staging structure, not persisted as plaintext alongside ciphertext.
- **Using Vitest `vi.stubEnv` alone for isolation test:** `stubEnv` modifies the current process's env; it does NOT prove that a separately-spawned agent process lacks the key. Child process test (D-12) is required.
- **Wrapping `ALTER TYPE ADD VALUE` in a transaction:** PostgreSQL does not allow `ADD VALUE` in a transaction block; this will cause migration to fail. The migration file must use the Drizzle breakpoint comment format (`--> statement-breakpoint`) but NOT be wrapped in a `BEGIN/COMMIT` block.
- **Regenerating already-approved scenarios:** D-09 requires only rejected scenarios are regenerated. Approved scenario IDs must be passed to the regeneration call to exclude them.
- **Allowing seal with fewer than 5 approved scenarios:** `sealVault()` must enforce `approvedCount >= 5` before calling encrypt.
- **Allowing the KEK to be passed between functions:** The KEK (`HOLDOUT_ENCRYPTION_KEY`) must only be read from `process.env` inside the vault seal/unseal function. It must never be returned to callers, logged, or included in Inngest step return values.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| AES-256-GCM encryption | Custom crypto wrapper | `node:crypto` built-in `createCipheriv`/`createDecipheriv` | GCM auth tag prevents tampering; no external dep surface |
| Schema validation for LLM output | Manual JSON parsing | Zod schema with `generateObject` | Type safety + automatic retry on parse failure by AI SDK |
| Cross-model diversity enforcement | Custom model-family check | Existing `enforceDiversity()` in `diversity.ts` | Already wired in gateway; calling with `stage: 'holdout'` is sufficient |
| Durable multi-step orchestration | Raw async chains | Inngest `step.run()` | Automatic retry, state persistence, replay-safe |
| Event-driven unseal trigger | Polling loop | Inngest event handler for `evolution_converged` | Event already in schema; no polling needed |

**Key insight:** The hardest part is NOT the crypto — AES-256-GCM is three function calls. The complexity is in the review state machine (pending_review → approved), the minimum-scenario guard before sealing, and the separation between the key-aware vault process and key-unaware agent processes.

## Common Pitfalls

### Pitfall 1: PostgreSQL ALTER TYPE ADD VALUE in Transaction
**What goes wrong:** Migration fails with "ALTER TYPE ... ADD VALUE cannot run inside a transaction block"
**Why it happens:** Drizzle Kit's default migration runner wraps each migration in a transaction; PostgreSQL cannot add enum values inside transactions.
**How to avoid:** Write migration 0004 manually (like Phase 3's 0003 migration) using the Drizzle breakpoint comment format. Do NOT use `db:generate` for enum extensions. Verify the migration file does not contain `BEGIN`/`COMMIT`.
**Warning signs:** Migration fails on first run with a clear PostgreSQL error about transactions.

### Pitfall 2: IV Reuse with AES-256-GCM
**What goes wrong:** Using the same IV+key combination more than once breaks GCM's security guarantees catastrophically (key stream reuse).
**Why it happens:** Manually managing IVs, forgetting to call `randomBytes(12)` freshly per encrypt call.
**How to avoid:** Generate IV inside the encrypt function every time. Never accept an IV as a parameter to `encryptWithDek`. Each `sealVault()` call generates a fresh DEK and IV.
**Warning signs:** Tests pass but IV is hardcoded or reused across test fixtures.

### Pitfall 3: Inngest Step Returns Containing Plaintext Scenarios
**What goes wrong:** Unsealed scenario JSON is stored in Inngest's step result cache, which may be logged or inspectable.
**Why it happens:** `step.run('unseal-vault', ...)` returns the decrypted JSON; Inngest persists step results for replay.
**How to avoid:** After unsealing, immediately pass scenarios to the evaluation step without returning them at the Inngest function level. Consider returning only evaluation results, not the raw scenario content, from the Inngest function.
**Warning signs:** Inngest Dev Server UI shows plaintext scenario content in step outputs.

### Pitfall 4: Missing Minimum Scenario Guard
**What goes wrong:** `sealVault()` called with 0-4 approved scenarios; vault sealed with insufficient coverage.
**Why it happens:** Guard not implemented in `sealVault()`; UI calls seal directly.
**How to avoid:** `sealVault()` must throw if `approved.length < 5`. Test this case explicitly.
**Warning signs:** No test covers the "seal with 4 scenarios" rejection path.

### Pitfall 5: DEK IV Storage Collision
**What goes wrong:** The `holdout_vault` table has one `iv` and one `auth_tag` column. Envelope encryption needs TWO IVs (one for data, one for DEK-encryption). If both are stored in the same column, decryption uses wrong IV.
**Why it happens:** The existing schema has `iv` and `auth_tag` (for data encryption). DEK encryption also produces an IV and auth tag. These must be stored separately.
**How to avoid:** Either (a) encode the DEK encryption metadata into the `encrypted_dek` text column as a compound value (`base64(iv):base64(authTag):base64(ciphertext)`), or (b) add `dek_iv` and `dek_auth_tag` columns in migration 0004. Option (a) avoids a schema change. Option (b) is cleaner but requires additional migration columns.
**Warning signs:** Decryption of DEK fails with authentication tag mismatch.

### Pitfall 6: Holdout Status Enum in Drizzle Schema vs DB
**What goes wrong:** Drizzle TypeScript enum definition and DB enum diverge; queries fail at runtime with type mismatch.
**Why it happens:** Migration extends the DB enum but `holdout.ts` schema file is not updated to include the new values.
**How to avoid:** After writing migration 0004, update `packages/shared/src/db/schema/holdout.ts` to extend `holdoutStatusEnum` with `'pending_review'` and `'approved'`.
**Warning signs:** TypeScript compiles but inserting `status: 'pending_review'` throws a Drizzle type error.

## Code Examples

Verified patterns from official sources and existing codebase:

### Envelope Encrypt/Decrypt (full round-trip)
```typescript
// Source: node:crypto official docs, confirmed working on Node v22.22.1
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';

const ALG = 'aes-256-gcm';

// Seal: generate DEK, encrypt payload, encrypt DEK with KEK
export function sealPayload(payload: string): {
  ciphertext: string; iv: string; authTag: string;
  encryptedDek: string; // compound: `${dekIv}:${dekAuthTag}:${dekCiphertext}` in base64
} {
  const kek = getKek();                    // throws if HOLDOUT_ENCRYPTION_KEY absent
  const dek = randomBytes(32);
  const iv = randomBytes(12);

  const cipher = createCipheriv(ALG, dek, iv);
  const enc = Buffer.concat([cipher.update(payload, 'utf8'), cipher.final()]);

  const dekIv = randomBytes(12);
  const dekCipher = createCipheriv(ALG, kek, dekIv);
  const encDek = Buffer.concat([dekCipher.update(dek), dekCipher.final()]);
  const dekAuthTag = dekCipher.getAuthTag();

  return {
    ciphertext: enc.toString('base64'),
    iv: iv.toString('base64'),
    authTag: cipher.getAuthTag().toString('base64'),
    // Compound DEK field avoids needing two extra DB columns
    encryptedDek: [
      dekIv.toString('base64'),
      dekAuthTag.toString('base64'),
      encDek.toString('base64'),
    ].join(':'),
  };
}

function getKek(): Buffer {
  const raw = process.env['HOLDOUT_ENCRYPTION_KEY'];
  if (!raw) throw new Error('HOLDOUT_ENCRYPTION_KEY is required for vault operations');
  return Buffer.from(raw, 'base64');
}
```

### Drizzle Holdout Schema Extension
```typescript
// Source: packages/shared/src/db/schema/holdout.ts — extend to add new status values
export const holdoutStatusEnum = pgEnum('holdout_status', [
  'pending_review',  // new
  'approved',        // new
  'sealed',
  'unsealed',
  'evaluated',
]);

export const holdoutVault = pgTable('holdout_vault', {
  id: uuid('id').primaryKey().defaultRandom(),
  seedId: uuid('seed_id').notNull().references(() => seeds.id),
  ciphertext: text('ciphertext').notNull(),
  encryptedDek: text('encrypted_dek').notNull(),
  iv: text('iv').notNull(),
  authTag: text('auth_tag').notNull(),
  status: holdoutStatusEnum('status').notNull().default('pending_review'),  // updated default
  results: jsonb('results'),          // new: evaluation results JSONB
  encryptedAt: timestamp('encrypted_at', { withTimezone: true }),           // was notNull().defaultNow() — now nullable until seal
  unsealedAt: timestamp('unsealed_at', { withTimezone: true }),
  evaluatedAt: timestamp('evaluated_at', { withTimezone: true }),           // new
});
```

**Note on schema change:** The existing `ciphertext`, `encryptedDek`, `iv`, `authTag` are `notNull()` in the current schema because the table was designed for already-sealed records. With `pending_review` and `approved` statuses preceding sealing, either (a) these columns must become nullable, or (b) the pre-seal state is stored elsewhere (e.g., a separate `holdout_drafts` table or in-memory). The simpler approach (a) requires a migration column alteration. Approach (b) avoids modifying existing notNull constraints. This is a Claude's Discretion item — but the planner must pick one. Recommendation: use approach (b) — store the pending/approved scenarios as a JSONB `draft_scenarios` column on `holdout_vault`, which gets cleared after sealing. This preserves all existing notNull constraints.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Direct encryption (no DEK) | Envelope encryption (DEK/KEK) | Standard in key management systems | Enables KEK rotation without re-encrypting all vaults |
| `aes-256-cbc` | `aes-256-gcm` | GCM became standard ~2015 | Authenticated encryption; auth tag prevents tampering without separate HMAC |
| Per-step env injection in Inngest | Dedicated function with env; agent workers lack env | Always been process-level | Isolation is architectural not runtime; agent-vs-vault function boundary is the security boundary |

**Deprecated/outdated:**
- `aes256` npm package: thin wrapper with no benefit, some CVEs. Use `node:crypto` directly.
- `crypto-ts`: browser-focused, different API surface. Not applicable for Node.js server-side encryption.
