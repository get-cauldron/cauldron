## Summary

Phase 6.1 is a wiring and invocation phase. Phases 1ŌĆō6 built every engine component: interview FSM, crystallizer, holdout vault, decomposition pipeline, bead scheduler, and the full Inngest execution path. This phase adds the surface layer: a CLI entrypoint in `packages/api/src/cli.ts` that imports and drives those components directly, and eight `.claude/skills/cauldron-*.md` files that invoke the CLI via `tsx`.

The engine's public API is already well-defined. `InterviewFSM` has `startOrResume()`, `submitAnswer()`, `generateSummary()`, `approveAndCrystallize()`. `runDecomposition()` is the single entry point for decompose ŌåÆ validate ŌåÆ persist ŌåÆ dispatch. `beadDispatchHandler` (wrapped as an Inngest function) drives bead execution. `configureSchedulerDeps()` and `configureVaultDeps()` wire runtime dependencies before Inngest handles events. The CLI's job is to construct these dependencies (db client, gateway, inngest client, logger) from env vars and config, then call the right engine function per sub-command.

The three interesting design challenges are: (1) how the CLI starts, serves, and re-uses the Inngest dev server across commands; (2) how the interview reads `.planning/` markdown files to avoid re-asking decided questions (brownfield interview extension); and (3) how the `selfBuild: true` flag gets added to `GatewayConfig` / `cauldron.config.ts` and propagated without creating cross-package coupling.

**Primary recommendation:** Keep the CLI as a thin orchestrator ŌĆö construct deps from env, call existing engine functions, write review files to disk. No new orchestration logic belongs in the CLI itself.

### Bootstrap Pattern

The `bootstrap.ts` module constructs all engine dependencies from environment variables and the project's `cauldron.config.ts`. This is the same pattern already used in the engine test setup:

```typescript
// Source: packages/engine/src/holdout/events.ts + packages/shared/src/db/client.ts
import { db } from '@cauldron/shared';
import { LLMGateway } from '@cauldron/engine';
import { loadConfig } from '@cauldron/engine/gateway';
import { Inngest } from 'inngest';
import pino from 'pino';

export async function bootstrap(projectRoot: string) {
  const config = await loadConfig(projectRoot);
  const logger = pino({ level: process.env['LOG_LEVEL'] ?? 'info' });
  const gateway = new LLMGateway({ db, config, logger });
  const inngestClient = new Inngest({ id: 'cauldron-engine' });
  return { db, gateway, inngest: inngestClient, logger, config };
}
```

**Critical:** `db` is a singleton from `@cauldron/shared` ŌĆö the existing `packages/shared/src/db/client.ts` already constructs and exports it from `DATABASE_URL`. The CLI imports it directly, exactly as the engine does.

### Inngest Serve Pattern for CLI

The `execute` command faces a challenge: Inngest requires a long-running HTTP server to receive events. The CLI needs to:
1. Call `configureSchedulerDeps()` and `configureVaultDeps()` with real deps
2. Register all Inngest functions
3. Serve the Inngest handler on a local port (e.g., 3001)
4. Send `bead.dispatch_requested` events to trigger the pipeline
5. Keep the process alive until all beads complete (or user kills it)

The existing engine already uses `new Inngest({ id: 'cauldron-engine' })` singleton in `packages/engine/src/holdout/events.ts`. The CLI re-uses the same singleton. The Inngest dev server (already in `docker-compose.yml`) connects to `http://host.docker.internal:3001/api/inngest` ŌĆö the CLI must serve on that port.

**Pattern from CLAUDE.md Stack Patterns:**
> "Use Hono (not Next.js Route Handlers) for the agent worker API surface" ŌĆö the CLI's Inngest HTTP handler should be served via Hono, not a raw Node.js http server.

```typescript
// Source: CLAUDE.md ┬¦Stack Patterns (MEDIUM confidence ŌĆö pattern, not verified code)
import { serve } from '@hono/node-server';
import { Hono } from 'hono';
import { serve as inngestServe } from 'inngest/hono';
import { functions } from '@cauldron/engine'; // array of InngestFunction

const app = new Hono();
app.on(['GET', 'PUT', 'POST'], '/api/inngest', ...inngestServe({ client: inngest, functions }));
serve({ fetch: app.fetch, port: 3001 });
```

**Note:** `inngest/hono` adapter requires `hono` to be added as a direct dependency to `packages/api`. This is the only new library needed beyond workspace packages.

### Dependency Wiring ŌĆö configureSchedulerDeps and configureVaultDeps

These two module-level config functions exist in the engine and MUST be called before Inngest processes any events. The CLI's bootstrap sequence is:

1. Load env vars (DATABASE_URL, HOLDOUT_ENCRYPTION_KEY, API keys)
2. Construct `{ db, gateway, inngest, logger }` via `bootstrap()`
3. Call `configureSchedulerDeps({ db, gateway, projectRoot })`
4. Call `configureVaultDeps({ db, gateway })`
5. Start Hono/Inngest HTTP server
6. Health check: verify DB connection + Inngest reachability (D-05)

### Health Check Pattern

```typescript
// Health check before any command (D-05)
async function healthCheck(db: DbClient): Promise<void> {
  try {
    await db.execute(sql`SELECT 1`);
  } catch {
    console.error('PostgreSQL not reachable. Run: docker compose up -d postgres');
    process.exit(1);
  }

  try {
    const res = await fetch('http://localhost:8288/v0/envs');  // Inngest dev server health
    if (!res.ok) throw new Error('Inngest not responding');
  } catch {
    console.error('Inngest dev server not reachable. Run: docker compose up -d inngest');
    process.exit(1);
  }
}
```

**Inngest dev server port:** 8288 (confirmed from `docker-compose.yml`). The engine handler serves on 3001 (Inngest dev server configured with `-u http://host.docker.internal:3001/api/inngest`).

### GSD Context Bridging Pattern

D-07 requires the interview to read `.planning/` files as prior context. The `InterviewFSM` already supports brownfield mode via `detectInterviewMode()` (uses git history). The interview command must:
1. Detect `selfBuild: true` in config (or check if `.planning/` exists)
2. Read and concatenate relevant `.planning/` artifacts: `PROJECT.md`, `REQUIREMENTS.md`, phase-specific `CONTEXT.md`
3. Inject this as an initial "prior context" message into the interview transcript before the first LLM call

The interview's `startOrResume()` accepts `{ mode, projectPath }`. The prior context injection needs a new optional parameter or a pre-populated transcript entry. Looking at `InterviewFSM.startOrResume()` ŌĆö it creates a new interview row with empty transcript. The cleanest approach is to pre-populate the `transcript` JSONB column with a system turn containing the GSD context before the FSM starts asking questions.

```typescript
// Conceptual pattern (Claude's discretion per D-07)
const priorContext = await readPlanningArtifacts(projectRoot); // reads .planning/ markdown
const interview = await fsm.startOrResume(projectId, {
  mode: 'brownfield',
  projectPath: projectRoot,
  priorContext, // new optional param ŌĆö injected as first transcript entry
});
```

This requires a minor extension to `InterviewFSM.startOrResume()` to accept and persist `priorContext` ŌĆö or it can be done by writing a "pre-answer" transcript entry immediately after creating the interview row, before `submitAnswer()` is called. The latter avoids touching the FSM internals.

### Engine Snapshot Mechanism

D-12 states "record current git commit hash and refuse to merge if the orchestrator detects the engine package changed since that hash." Implementation approach:

```typescript
// Simplest viable implementation ŌĆö git commit hash at run start
import { execSync } from 'node:child_process';

function captureEngineSnapshot(projectRoot: string): string {
  return execSync('git rev-parse HEAD', { cwd: projectRoot, encoding: 'utf-8' }).trim();
}

function detectEngineChange(snapshotHash: string, projectRoot: string): boolean {
  const currentHash = execSync('git rev-parse HEAD', { cwd: projectRoot, encoding: 'utf-8' }).trim();
  return currentHash !== snapshotHash;
}
```

Store the snapshot hash in `project_snapshots.state` JSONB (the existing snapshot table) with a `selfBuildSnapshotHash` key. The merge queue checks this before merging any bead that touches `packages/engine/`.

### Migration Review Gate (D-13)

The `MergeQueue` already has post-merge test failure detection (Phase 6 D-16). The migration review gate adds: before merging any bead whose diff contains files matching `packages/shared/src/db/migrations/**`, emit a `merge_escalation_needed` event and write the migration diff to a review file. The merge is blocked until `cauldron resolve <bead-id>` approves it.

This extends the existing escalation path ŌĆö `MergeQueue` already escalates on LLM conflict confidence < threshold. Migration detection is a content-based escalation rule added to the pre-merge check step.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Inngest HTTP handler | Custom Node.js `http.createServer` | `inngest/hono` adapter + Hono | Inngest SDK ships adapters for all major Node frameworks; Hono is already the designated standalone API framework per CLAUDE.md |
| DB client construction | New drizzle setup | `import { db } from '@cauldron/shared'` | Singleton already constructed from `DATABASE_URL`; duplicating it creates two connection pools |
| Argument parsing | Custom `process.argv` string splitting | `node:util` `parseArgs` (Node 18+, no install) | Built-in since Node 18; handles flags, positional args, `--` separator without dependencies |
| Status table rendering | Custom padding/alignment | `console.table()` | Node 22 native; sufficient for a 5-column terminal table |
| Log tailing | Custom `LISTEN`/`NOTIFY` WebSocket | `db.execute(sql`LISTEN cauldron_events`)` or polling events table | Postgres LISTEN/NOTIFY is available; polling the events table every 2s is simpler and avoids a persistent connection during skills |
| File YAML serialization | Custom format | `node:yaml` (Node 22) or `JSON.stringify` + `.yaml` extension | Skills write/read review files once; minimal serialization needed; avoid external deps |

**Key insight:** Every domain problem (LLM calls, DB ops, Inngest dispatch, event sourcing) already has a solution in the existing codebase. The CLI is pure wiring.

## Common Pitfalls

### Pitfall 1: Inngest Client Singleton Collision
**What goes wrong:** Creating a `new Inngest({ id: 'cauldron-engine' })` in the CLI that is separate from the one in `packages/engine/src/holdout/events.ts`. Two clients with the same app ID fight over function registration.
**Why it happens:** The CLI imports engine modules that have their own module-level Inngest singleton.
**How to avoid:** The CLI must NOT create its own Inngest client. Import and re-use the `inngest` singleton exported from `packages/engine/src/holdout/events.ts`. All engine modules use the same singleton because they import from each other ŌĆö the CLI just joins that import chain.
**Warning signs:** Inngest dev server shows duplicate function registrations, or events are processed twice.

### Pitfall 2: configureSchedulerDeps / configureVaultDeps Called After Inngest Starts
**What goes wrong:** An Inngest event fires before the dependency configuration functions have been called. `getSchedulerDeps()` throws: "Scheduler dependencies not configured."
**Why it happens:** Async initialization races with Inngest event processing if the HTTP server starts before `configureSchedulerDeps()`.
**How to avoid:** The call sequence must be: (1) configure deps, (2) start HTTP server. Never start serving until deps are configured.
**Warning signs:** `Error: Scheduler dependencies not configured` in Inngest function logs on first event.

### Pitfall 3: DATABASE_URL Not Set at Import Time
**What goes wrong:** `packages/shared/src/db/client.ts` throws immediately at module import if `DATABASE_URL` is unset. This crashes the CLI before any code runs.
**Why it happens:** The shared client uses a module-level `const connectionString = process.env['DATABASE_URL']` with a `throw` ŌĆö no lazy init.
**How to avoid:** Ensure `DATABASE_URL` is set in the shell environment (or `.env` file with `dotenv/config` auto-load, which `client.ts` already imports via `import 'dotenv/config'`). The CLI should document required env vars in the health check output.
**Warning signs:** `Error: DATABASE_URL environment variable is required` on any `tsx packages/api/src/cli.ts` invocation.

### Pitfall 4: Inngest Dev Server Not Connected When execute Runs
**What goes wrong:** `execute` dispatches events via Inngest but no worker is running to process them. Events queue indefinitely. Status shows all beads 'pending' forever.
**Why it happens:** The CLI starts its HTTP server on port 3001 AFTER sending events, or the Inngest dev server container is not running, so events have no handler.
**How to avoid:** Health check (D-05) must verify Inngest dev server at `http://localhost:8288` before any `execute` invocation. The CLI's HTTP server must be running before dispatching any events.
**Warning signs:** Beads remain 'pending' after `/cauldron:execute`. Inngest dashboard (port 8288) shows events with no function executions.

### Pitfall 5: GatewayConfig vs ProjectSettings Confusion for selfBuild
**What goes wrong:** `selfBuild` flag placed in `ProjectSettings` (DB-stored per-project) instead of `GatewayConfig` (file-based per-project). This means it's mutable and doesn't survive DB wipes.
**Why it happens:** `ProjectSettings` is in `packages/shared` (accessible to all layers), `GatewayConfig` is in `packages/engine` ŌĆö the engine layer feels "too deep" for a project-level flag.
**How to avoid:** `GatewayConfig` is loaded from `cauldron.config.ts` at startup and is the source of truth for project-level configuration. `ProjectSettings` is for runtime overrides stored in DB. `selfBuild` is a development-time flag ŌĆö it belongs in `cauldron.config.ts` ŌåÆ `GatewayConfig`. The `loadConfig()` function already handles this.
**Warning signs:** `selfBuild` flag is set in DB but ignored after a `db:migrate` wipes the projects table.

### Pitfall 6: Interview Prior-Context Injection Confuses the Scoring FSM
**What goes wrong:** Injecting GSD planning context as a raw transcript turn causes the ambiguity scorer to treat it as a user answer, producing wildly wrong scores and premature crystallization.
**Why it happens:** The interview FSM's `submitAnswer()` scores every turn including the first. A 5000-token planning document injected as a "user answer" breaks the score calibration.
**How to avoid:** Inject prior context as a special `role: 'system'` or `role: 'assistant'` turn that the scorer ignores. Alternatively, add a `priorContext` field to the interview row (JSONB) that the FSM reads as background but does not score. The cleanest approach: write a `type: 'prior_context'` turn to the transcript with `role: 'system'` ŌĆö the scorer checks for this turn type and skips scoring it. Requires a small FSM extension.
**Warning signs:** Interview crystallizes after 0ŌĆō1 turns because ambiguity score is already 0.0 (the LLM concluded all context is clear from the planning docs).

### Pitfall 7: tsx Not on PATH in Skill Invocation
**What goes wrong:** `.claude/skills/cauldron-*.md` calls `tsx packages/api/src/cli.ts` but `tsx` is a root `devDependency` not globally installed. Claude Code's shell may not have the project's `node_modules/.bin` on PATH.
**Why it happens:** `tsx` is installed in `node_modules/.bin/tsx` by pnpm. The shell may not have `./node_modules/.bin` on PATH unless `pnpm exec` or `npx` is used.
**How to avoid:** Skills invoke the CLI via `pnpm exec tsx packages/api/src/cli.ts <command>` or `node --import tsx/esm packages/api/src/cli.ts <command>`. The former is preferred since pnpm is already the workspace package manager. Alternatively, add a `cauldron` script entry to the root `package.json` scripts.
**Warning signs:** `command not found: tsx` in Claude Code skill execution.

### Pitfall 8: Engine Snapshot Does Not Cover Compiled Output
**What goes wrong:** The git commit hash snapshot catches source changes but not compiled dist artifacts. If a build step produces stale `dist/` output, agents may work against a different version than what the hash suggests.
**Why it happens:** `cauldron.config.ts` is loaded via dynamic `import()` which resolves the source `.ts` file directly (via `tsx`) ŌĆö no compiled dist involved. But if any part of the pipeline accidentally uses compiled dist output, the hash check gives false confidence.
**How to avoid:** The entire engine runs as TypeScript source via `tsx` (confirmed by engine and API `package.json` scripts). No dist artifacts are involved in the CLI execution path. The git hash snapshot is accurate. Document this assumption in the `selfBuild` config.
**Warning signs:** Any `import` call resolving to a `.js` file in `dist/` instead of `.ts` in `src/`.

## Open Questions

1. **Where does the Inngest HTTP server run during `execute`?**
   - What we know: The CLI needs to serve Inngest on port 3001. This means `execute` is a long-running process, not a quick command.
   - What's unclear: Should the CLI start the HTTP server in a background child process and return immediately, or should it block while displaying a live status view?
   - Recommendation: Start the Inngest server in-process (blocking). `/cauldron:execute` blocks until all beads are complete (or user Ctrl-C). `/cauldron:status` is for a separate terminal window. This is the simplest model ŌĆö consistent with D-03 (fire-and-poll applies to the skill invoking execute, not to execute itself blocking).

2. **Does the `inngest` singleton from `holdout/events.ts` need to be re-exported from the engine index?**
   - What we know: `packages/engine/src/index.ts` exports gateway, interview, and decomposition but not holdout. The CLI needs the singleton.
   - What's unclear: Is the engine's index.ts the right place to re-export it, or should the CLI import directly from the deep path `@cauldron/engine/holdout`?
   - Recommendation: Add `export { inngest } from './holdout/events.js'` to `packages/engine/src/index.ts`. The singleton must be shared ŌĆö exporting it makes the contract explicit.

3. **How does the dry run on the rename tool (D-09) get set up?**
   - What we know: It must be a separate git repo to prove external project support. The CLI must run against it with a different `--project-root`.
   - What's unclear: Does the plan include creating that repo, or is it assumed to pre-exist?
   - Recommendation: The plan should include a Wave that creates a minimal file-rename CLI project as a fixture repo (one `package.json`, a `src/` folder, no existing tests). This is the dry run target. It needs its own `cauldron.config.ts` with `selfBuild: false`.
