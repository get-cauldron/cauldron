# Phase 6.1: Dogfooding Transition - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the invocation surface and bridging infrastructure that lets Claude Code drive Cauldron's engine to plan and execute subsequent phases (7-9) on Cauldron's own codebase. Delivers: CLI entrypoint in `packages/api`, project-local Claude Code skills per pipeline stage, GSD-to-Cauldron context bridging, self-build safety mode, and a dry-run validation on the v1 test case (CLI renaming tool in a separate repo). This phase does NOT implement the evolutionary loop (Phase 7), dashboard (Phase 8), or CLI distribution (Phase 9).

</domain>

<decisions>
## Implementation Decisions

### Invocation Surface
- **D-01:** Separate project-local Claude Code skills per pipeline stage — `/cauldron:interview`, `/cauldron:crystallize`, `/cauldron:decompose`, `/cauldron:execute`, `/cauldron:status`, `/cauldron:kill`, `/cauldron:seal`, `/cauldron:resolve`. Skills live in `.claude/skills/cauldron-*.md`, versioned with the repo.
- **D-02:** Skills invoke a CLI entrypoint in `packages/api/src/cli.ts` via tsx. CLI imports engine modules directly. In-process execution — no separate Inngest process needed for the skill itself. Inngest handles bead dispatch internally within the engine.
- **D-03:** Fire-and-poll execution model for long-running operations. `/cauldron:execute` kicks off bead dispatch and returns immediately. `/cauldron:status` polls progress. Execution runs in background via Inngest.
- **D-04:** Pipeline state stored in PostgreSQL (existing DB). `/cauldron:status` queries the same tables the engine writes to: beads, bead_edges, events, seeds.
- **D-05:** Skills include health checks — verify DB and Inngest are reachable before running. If not, prompt user to start Docker Compose.

### Bootstrapping Scope
- **D-06:** Full pipeline path minus evolutionary loop: interview → crystallize seed → generate holdouts → seal → decompose → execute beads → merge. One pass through the pipeline per phase. Manual iteration if needed.
- **D-07:** Bootstrap from GSD context — interview reads `.planning/` directory as prior requirements context. CONTEXT.md decisions become interview pre-answers. REQUIREMENTS.md provides acceptance criteria. Interview focuses on gaps only, not re-asking decided questions.
- **D-08:** Include holdout test generation + encryption in the dogfood path. Proves the encryption pipeline end-to-end. Unseal manually during dogfooding since the evo loop doesn't exist yet.
- **D-09:** Dry run on trivial project first — validate the pipeline on the v1 test case (CLI bulk file renaming tool) in a separate repo before attempting self-build on Phase 7.
- **D-10:** Seed acceptance criteria map back to GSD requirement IDs (EVOL-01, EVOL-02, etc.) for traceability from seed → requirements → roadmap.

### Skill Rollout Strategy
- **D-11:** CLI entrypoint first, skills wrap it. Build order: (1) `packages/api/src/cli.ts` core driver, (2) `/cauldron:status` (read-only, simplest), (3) `/cauldron:interview`, (4) `/cauldron:crystallize`, (5) `/cauldron:decompose`, (6) `/cauldron:execute`, (7) `/cauldron:kill`, (8) `/cauldron:seal`, (9) `/cauldron:resolve`. Dry run after step 6.

### Self-Referential Safety
- **D-12:** Engine snapshot at run start — before pipeline execution begins, the engine version is captured. Agents work in worktrees, orchestrator runs against the snapshot. Merged results don't affect the running engine until the next run.
- **D-13:** Migrations allowed with review gate — agents can create migration files, but they're flagged for human review before running. No auto-migrate during execution.
- **D-14:** Auto-revert on post-merge test failure — revert the merge, mark bead as failed, surface error context. Keeps main stable. (Consistent with Phase 6 D-16.)
- **D-15:** Explicit `selfBuild: true` config flag activates extra safety guardrails: engine snapshot, migration review gates, stricter merge tests. Visible and toggleable.

### Human Steering Without Dashboard
- **D-16:** File-based seed review — interview outputs a seed YAML draft to disk. User edits in their editor. `/cauldron:crystallize` finalizes. Same edit-then-approve flow as Phase 3, file-based.
- **D-17:** File-based holdout review — same pattern as seed review. Scenarios written to a review file. User approves/rejects/edits individual scenarios. `/cauldron:seal` finalizes.
- **D-18:** Status table + optional log tail — `/cauldron:status` shows terminal table (bead name, status, agent, duration). Pass `--logs` to tail raw events. Like `kubectl get pods` for beads.
- **D-19:** Merge conflict escalation via status flag + file diff — `/cauldron:status` shows 'NEEDS REVIEW'. Conflict file written with both sides. User resolves in editor, then `/cauldron:resolve <bead-id>`.
- **D-20:** Kill only for bead intervention — `/cauldron:kill <bead-id>` to abort a stuck bead. No mid-run hinting. Keep it simple.

### Error Recovery
- **D-21:** Resume from DB state — all bead statuses are in PostgreSQL. `/cauldron:execute --resume` checks DB, skips completed beads, re-dispatches failed/pending ones. DB is the source of truth.

### Claude's Discretion
- CLI argument parsing and command routing implementation
- Skill markdown content and triggering descriptions
- Health check implementation details (connection timeouts, error messages)
- Engine snapshot mechanism (git tag, copy, or process isolation)
- Status table formatting and column layout
- Log tail implementation (LISTEN/NOTIFY or polling)
- GSD artifact parsing logic within the interview
- How selfBuild flag is read and propagated through the engine
- Dry run project setup and teardown

</decisions>

## Existing Code Insights

### Reusable Assets
- **Full engine pipeline** (`packages/engine/src/`): interview, decomposition, execution, holdout, gateway, intelligence — all built in Phases 1-6. The CLI just needs to wire these together.
- **Event store** (`packages/shared/src/db/event-store.ts`): Append events for all lifecycle tracking — status skill can query this.
- **Inngest handler patterns** (Phase 4, 6): `createFunction({ id, triggers })` established pattern for bead dispatch, holdout operations.
- **Docker Compose** (`docker-compose.yml`): PostgreSQL, Redis, Inngest dev server already configured from Phase 1.

### Established Patterns
- Drizzle ORM for all DB operations
- `.js` extensions on all relative TypeScript imports
- Zod schemas for structured LLM output
- Event sourcing: all state changes as immutable events
- Inngest v4 `createFunction` with `step.run()` and `step.waitForEvent()`
- Integration tests against real PostgreSQL (no mocks)

### Integration Points
- `packages/api/src/cli.ts` — new entrypoint importing from `@cauldron/engine`
- `.claude/skills/cauldron-*.md` — new skill files invoking CLI via tsx
- `cauldron.config.ts` — may need `selfBuild` flag addition
- Interview brownfield mode — extend to read `.planning/` as prior context
- Inngest `bead.dispatch_requested` event — already exists, skills just trigger the pipeline that emits it

</code_context>

<specifics>
## Specific Ideas

- The CLI renaming tool dry run should be a separate git repo to prove Cauldron works on external projects, not just itself
- Interview reading `.planning/` is a natural extension of brownfield mode — the knowledge graph already indexes the codebase; `.planning/` markdown is just additional context
- The skill build order is designed so each skill is independently testable: status (read-only) validates DB connectivity, interview validates LLM + DB, execute validates the full pipeline
- Engine snapshot could be as simple as recording the current git commit hash and refusing to merge if the orchestrator detects the engine package changed since that hash
- The `selfBuild` flag is a project-level config (in `cauldron.config.ts` or project settings in DB), not a per-run flag — consistent with per-project model overrides

</specifics>
