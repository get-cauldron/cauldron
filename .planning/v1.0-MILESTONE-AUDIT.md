# Milestone v1.0 Audit Report

**Milestone:** Cauldron v1.0
**Audited:** 2026-03-27
**Status:** GAPS FOUND — Critical cross-phase integration breaks prevent end-to-end pipeline execution

## Executive Summary

All 11 phases are individually complete with passing verifications, tests, and typechecks. Each phase's code is substantive, well-tested, and architecturally sound. **However, the integration layer has critical gaps**: the tRPC web layer and Inngest serve configuration do not properly wire engine functions into production paths, breaking 3 of 7 core E2E flows and leaving ~50 requirements unreachable in production despite being fully implemented in isolation.

**Root causes (2):**
1. **Engine Inngest functions have no HTTP serve endpoint.** The `cauldron-engine` client registers 5 functions but no route serves them — Inngest cannot deliver events.
2. **Phase 9 tRPC refactoring created stub mutations.** CLI commands were refactored from direct engine calls to tRPC calls, but several tRPC mutations (sendAnswer, sealHoldouts, triggerDecomposition) record DB events without invoking the engine functions that do the actual work.

**Note:** Phase 6.2 demonstrated the full pipeline working end-to-end via direct engine calls (6 beads completed). The integration breaks were introduced by the Phase 9 tRPC refactoring, which replaced direct engine calls with tRPC mutations that don't properly delegate to the engine.

## Cross-Phase Integration Results

### Systemic Root Cause 1: Engine Inngest Functions Not Served

The `cauldron-engine` Inngest client (`packages/engine/src/holdout/events.ts`) registers 5 functions:
- `handleBeadDispatchRequested`
- `handleBeadCompleted`
- `handleMergeRequested`
- `handleEvolutionConverged`
- `handleEvolutionStarted`

**No HTTP serve endpoint exists for this client.** The web's `/api/inngest` route only serves the `cauldron-web` client with `pipelineTriggerFunction`. Inngest cannot deliver events to any engine function.

**Impact:** All of EXEC (9 reqs), CODE (4 reqs), TEST (6 reqs), DAG-06–09 (4 reqs), EVOL (12 reqs), HOLD-05–08 (4 reqs) = **39 requirements unreachable** in production despite code being complete.

### Systemic Root Cause 2: tRPC Mutations Don't Invoke Engine

Phase 9 refactored all CLI commands from direct engine imports to tRPC calls. Several tRPC mutations are stubs:

| tRPC Mutation | What It Does | What It Should Do |
|---------------|-------------|-------------------|
| `interview.sendAnswer` | Appends partial user turn to DB | Call `InterviewFSM.submitAnswer()` for scoring + next question |
| `interview.sealHoldouts` | Sets `status='sealed'` in DB | Call `sealVault()` for AES-256-GCM encryption |
| `execution.triggerDecomposition` | Appends `decomposition_started` event | Call `runDecomposition()` or send Inngest event that triggers it |
| `execution.triggerExecution` | Appends `execution_started` event | Ensure engine Inngest functions are served to handle dispatch |

**Impact:** Interview, holdout sealing, and decomposition are broken end-to-end.

### Additional Integration Gap: configureEvolutionDeps Missing

`bootstrap.ts` calls `configureSchedulerDeps()` and `configureVaultDeps()` but not `configureEvolutionDeps()`. Any `evolution_started` event will throw "Evolution dependencies not configured".

## What Went Right

1. **Every phase's code is individually excellent.** 11/11 phases pass verification with substantive implementations, comprehensive tests (700+ tests total), and clean typechecks.
2. **The architecture is sound.** Event sourcing, immutable seeds, envelope encryption, Kahn's cycle detection, optimistic concurrency — all implemented correctly.
3. **Phase 6.2 proved the pipeline works.** The dogfooding validation demonstrated end-to-end execution with real LLM calls, seed crystallization, holdout sealing, and 6 bead completions.
4. **Read paths are fully wired.** Dashboard and CLI correctly display all data via tRPC queries.

## What Went Wrong

1. **Phase 9 tRPC refactoring broke the write path.** Direct engine calls were replaced with tRPC mutations that are effectively stubs — they record events but don't invoke engine logic.
2. **Dual Inngest clients were never unified.** `cauldron-engine` and `cauldron-web` are separate Inngest app IDs; only `cauldron-web` has a serve endpoint.
3. **Phase verifications checked code correctness but not cross-phase wiring.** Each phase verified that its own functions work, but no verification checked whether the tRPC layer actually calls those functions.
