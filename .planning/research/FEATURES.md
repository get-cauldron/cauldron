# Feature Research

**Domain:** AI-powered software development platform / autonomous software factory
**Researched:** 2026-03-25
**Confidence:** MEDIUM-HIGH (competitive features from current sources; Cauldron-specific novel features inferred from architecture)

---

## Feature Landscape

### Table Stakes (Users Expect These)

Features that any serious AI development platform must have. Missing these signals the product is incomplete or immature.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Natural language → code generation | Every competitor does this; it's the entry-level promise | MEDIUM | The commodity baseline. Must be end-to-end, not just snippet generation. |
| Multi-file / full-project context | Single-file AI suggestions feel toy-like. Cursor's whole-codebase indexing set this expectation. | HIGH | Requires brownfield codebase mapping (knowledge graph, sub-ms queries). Embedding search degrades at scale — must combine with grep/graph retrieval. |
| Self-healing / autonomous error correction | Devin and Replit Agent 3 both loop on test failures and fix them without user prompts. Users now expect this. | HIGH | Must read error logs, iterate on code, rerun tests. "Write code and stop" is no longer sufficient. |
| Sandboxed execution environment | Every platform (Devin, Replit, Bolt.new) runs code in isolated environments. Security non-negotiable. | HIGH | Isolated execution prevents agents from harming host systems. Cauldron runs per-bead agents, so isolation is per-bead. |
| Real-time progress visibility | Replit Agent 3 and Devin both stream logs. Users left in the dark abandon the session. | MEDIUM | Streaming logs + diffs during execution. DAG visualization is Cauldron's differentiating layer on top of this base. |
| Git integration (PRs, diffs, branches) | Devin creates PRs natively. Cursor 2.0 uses isolated git worktrees. GitHub Copilot Workspace was built around this. | MEDIUM | Git-push triggered runs, PR creation, branch isolation per parallel execution track. |
| Model selection / provider flexibility | OpenHands, Cursor, and Copilot all let users select/swap models. Lock-in to a single provider is a dealbreaker for professional teams. | MEDIUM | Vercel AI SDK provides the abstraction layer. Opinionated defaults with per-project overrides. |
| Iterative refinement via chat | v0, Bolt.new, and Lovable all support chat-driven iteration after initial generation. Users expect conversational correction. | MEDIUM | The Socratic interview is pre-generation; post-generation chat iteration is a separate table-stakes surface. |
| Project persistence across sessions | Replit Agent 3 runs for 200 minutes. Devin maintains session state. Users expect work to survive browser close. | HIGH | Event-store or persistent checkpointing required. Ouroboros EventStore pattern covers this. |
| Test generation | Devin writes tests, Copilot Workspace synthesizes test cases, OpenHands fixes test failures. Lack of tests signals shallow output. | HIGH | Testing cube (unit + integration + E2E) is Cauldron's position. Generate all three, not just unit tests. |
| CLI interface | Agentic coding is increasingly CLI-first (Claude Code, OpenHands CLI, Devin CLI). Developers expect terminal access. | LOW | Parallel surface to web dashboard. Git-push trigger is a sub-feature. |

### Differentiators (Competitive Advantage)

Features that no current platform has fully implemented, or where Cauldron's architecture creates a structural advantage.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Socratic interview with deterministic ambiguity scoring | Every other platform jumps straight to generation from a single prompt. Cauldron's Ouroboros-derived interview surfaces hidden assumptions before a line of code is written. Ambiguity ≤ 0.2 gate prevents premature crystallization. | HIGH | Multi-perspective question generation (researcher, simplifier, architect, breadth-keeper, seed-closer). No competitor implements this; AWS Kiro gets closest with structured requirements.md but lacks the scoring gate. |
| Immutable seed spec with full lineage tracking | Seeds never mutate. Evolution creates new seeds. This gives clean version history, rollback capability, and deterministic re-runs. No competitor tracks specification lineage this way. | HIGH | YAML-format seed (goal, constraints, acceptance criteria, ontology schema, evaluation principles, exit conditions). The immutability is the differentiator — not just writing a spec, but locking it cryptographically as a foundation for one execution tree. |
| Cross-model holdout tests (encrypted, adversarial) | Holdout tests generated by a different LLM family than the implementer, encrypted at rest, inaccessible to implementation agents. Prevents correlated blind spots and model gaming. No competitor does this. | HIGH | Holdout unsealing only after evolutionary convergence. Human review before encryption. This is the single largest architectural differentiator: adversarial cross-model validation built into the pipeline, not bolted on. |
| DAG-based bead decomposition with parallel-by-default execution | Cursor 2.0 runs up to 8 parallel agents in isolated VMs — but with no structured dependency graph. Cauldron's molecules/beads hierarchy enforces explicit dependency edges (blocks, parent-child, conditional-blocks, waits-for), enabling safe parallelism at any scale. | HIGH | Beads sized to fit one fresh context window per commercial model. Atomic claim prevents race conditions. Fan-out/fan-in synchronization gates. |
| Fresh context window per bead (context rot prevention) | Context rot is an acknowledged problem for long-running agents. Cursor 2.0 and Replit Agent 3 both deal with it via ad-hoc compaction. Cauldron structurally prevents it: each bead starts fresh. | HIGH | GSD-derived principle. Not a workaround but an architectural constraint. Every task fits in ~200k tokens with room to implement. |
| Evolutionary loop with convergence detection and lateral thinking | The evolutionary loop (post-execution evaluation → new seed → re-execute) is unique to Cauldron/Ouroboros. No commercial platform has this. Devin re-plans dynamically but within a single seed; it doesn't formally evolve the spec. | HIGH | Convergence criteria: ontology stability, stagnation, oscillation, repetitive feedback, hard cap. Lateral thinking personas (contrarian, hacker, simplifier, researcher, architect) activate on stagnation to escape local maxima. |
| DAG visualization as first-class UI surface | Langflow provides visual DAG editing for workflow design. No platform provides live DAG execution visualization showing which beads are active, blocked, or completed in real time. | MEDIUM | HZD Cauldron aesthetic creates visual identity. Dark metallic + teal energy conduits = instantly recognizable product. |
| Spec-as-truth philosophy with escalation (not full autonomy) | The "autonomous employee" framing (Devin) removes human judgment. The "copilot" framing (Cursor, Copilot) requires constant human steering. Cauldron operates autonomously but escalates when convergence looks unlikely. The human steers at key decision points, not every step. | HIGH | Structured escalation mechanism: approval gates at seed crystallization, holdout test review, convergence failure. Confidence-based routing to human when stagnation is detected. |
| Testing cube (unit + integration + E2E with equal depth) | The testing pyramid (lots of unit, few E2E) is the industry default because E2E tests are expensive for humans. AI agents have no patience constraint: E2E should match unit depth. No platform currently enforces this. | HIGH | AI generates thorough tests at all three levels per feature. Encrypted holdouts provide a fourth adversarial layer beyond the testing cube. |
| Brownfield codebase knowledge graph (sub-ms queries) | Most platforms use embedding-based retrieval which degrades at scale. Cauldron uses codebase-memory-mcp (knowledge graph + fast indexing) as an agent context layer, not a standalone tool. | HIGH | Incremental re-indexing as agents modify code. Knowledge graph enables dependency-aware decomposition during bead planning. |

### Anti-Features (Deliberately Not Built)

Features that seem obviously desirable but create structural problems for Cauldron's pipeline or user model.

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Mutable specs / in-place requirements editing | Users want to "just update" the spec when they change their mind | Mutating a seed breaks lineage tracking, creates ambiguous execution history, and allows spec drift during active runs. The entire evolutionary loop depends on spec immutability for clean lineage. | Create a new seed through the interview process. Evolution creates new seeds from old ones — this IS the update mechanism. |
| Credit-burn model (charge per AI error correction loop) | Platforms like Bolt and Lovable use per-token or per-credit billing that charges users when the AI fixes its own bugs | Perverse incentive: platform profits from its own mistakes. Creates user resentment (Lovable users reported burning 400 credits/hour in error loops). Trust destruction. | Flat-rate or time-based pricing. Errors are the platform's cost of doing business, not a billable event. |
| Fully autonomous "no human needed" mode | "Just build it while I sleep" is the dream pitch. Devin leans into this. | AI has correlated blind spots at current capability levels. Without human approval gates (holdout test review, convergence escalation), the system optimizes toward spec compliance rather than user intent. Also a liability concern. | Structured autonomy with escalation. Runs without babysitting, but humans approve holdout tests, review seed crystallization, and get notified on convergence failure. |
| Real-time collaborative multi-user editing | Google Docs-style multi-user presence on the same project | Concurrent modification of DAG state and seed specs creates conflict resolution problems that are orthogonal to the core pipeline. Adds significant infrastructure complexity. | v2+ concern. Single-tenant per-project in v1. Async review/approval is sufficient for team use in v1. |
| Deployment pipeline integration (CI/CD, cloud infra) | Full end-to-end from prompt to deployed production | Deployment adds cloud provider-specific complexity (IAM, networking, scaling) that is orthogonal to proving the development pipeline works. | v2+ concern per PROJECT.md. Prove the development pipeline first. |
| Drag-and-drop visual workflow builder | Langflow-style canvas for building agent workflows | Cauldron's pipeline structure is fixed (interview → seed → decompose → execute → evaluate). A drag-and-drop builder implies arbitrary pipeline construction, which conflicts with the spec-as-truth philosophy. Also creates a "WYSIWYG trap" where users design around the tool's UI rather than around their requirements. | Configuration-driven customization (model selection per stage, convergence thresholds). Visual output (DAG) but not visual input. |
| Streaming "vibe coding" output (watch it type code live) | v0, Bolt, and Lovable stream code generation as entertainment/engagement | Creates the illusion of fast progress while masking output quality. Users fixate on generation speed rather than evaluating whether output meets requirements. Encourages prompt-first thinking rather than spec-first. | Stream execution status and diff summaries. Detailed logs available on demand. Focus engagement on spec completeness and test results, not token throughput. |
| Single-model vendor lock-in | "Best results with Claude" or "optimized for GPT-4o" marketing | Model families have correlated blind spots. Cauldron's holdout testing requires cross-model diversity by design. Lock-in to a single provider eliminates the adversarial cross-model guarantee. | Vercel AI SDK abstraction layer. Opinionated defaults (e.g., "use Claude for implementation, GPT-4o for holdout generation") with per-project overrides. |
| Context window compaction via summarization | Claude Code's auto-compact approach, Cursor's long-context tricks | Summarization loses precision. Compacted context loses implementation details that agents need to stay consistent. The "brownfield tax" compounds when agents work from summaries. | Fresh context per bead. Structural prevention of context rot, not after-the-fact remediation. |

## Feature Dependencies

```
Socratic Interview
    └──produces──> Immutable Seed
                       └──requires──> Holdout Test Generation (cross-model)
                       └──requires──> Bead DAG Decomposition
                                          └──requires──> Parallel Execution Engine
                                                             └──requires──> Atomic Bead Claiming
                                                             └──requires──> Sandboxed Execution
                                                             └──requires──> Brownfield Knowledge Graph (for context loading)

Holdout Test Generation
    └──requires──> Cross-Model Diversity (different provider than implementer)
    └──requires──> Encryption at Rest (keys inaccessible to implementation agents)
    └──followed-by──> Holdout Unsealing (post-convergence only)

Parallel Execution Engine
    └──produces──> Execution Results
                       └──requires──> Testing Cube (unit + integration + E2E)
                       └──feeds──> Evolutionary Loop Evaluation

Evolutionary Loop Evaluation
    └──requires──> Convergence Detection (ontology stability, stagnation, oscillation)
    └──produces──> New Immutable Seed (never mutates existing seed)
    └──may-trigger──> Lateral Thinking Personas (on stagnation)
    └──may-trigger──> Human Escalation (on convergence failure)

Holdout Unsealing
    └──requires──> Evolutionary Convergence
    └──produces──> Final Validation Result

DAG Visualization (UI)
    └──requires──> Bead DAG Decomposition
    └──requires──> Real-Time Execution Status
    └──enhances──> Parallel Execution Engine (observability)

Brownfield Knowledge Graph
    └──requires──> Codebase Indexing (sub-ms queries)
    └──enhances──> Bead DAG Decomposition (dependency-aware)
    └──enhances──> Agent Context Loading (per-bead)
```

### Dependency Notes

- **Immutable Seed requires Socratic Interview:** The seed cannot be generated until ambiguity score ≤ 0.2. The interview IS the gate.
- **Holdout Test Generation requires Cross-Model Diversity:** Using the same LLM family as the implementer defeats the adversarial guarantee. This is not a nice-to-have; it is the mechanism that makes holdouts meaningful.
- **Evolutionary Loop requires Testing Cube:** The loop can only evaluate "did we meet the goal, not just the spec?" if comprehensive test coverage exists to detect the difference.
- **Holdout Unsealing conflicts with mid-execution access:** Implementation agents must never access holdout tests. Encryption at rest enforces this. The conflict is intentional.
- **Fresh Context per Bead requires Bead DAG Decomposition:** You cannot have context isolation without knowing where the task boundaries are. The DAG defines the boundaries.
- **Lateral Thinking Personas enhance Evolutionary Loop:** Not a separate feature — a mode activation within the loop triggered by stagnation detection.

## MVP Definition

### Launch With (v1)

The minimum to demonstrate the full pipeline end-to-end on the v1 test case (CLI bulk file renaming tool).

- [ ] Socratic interview with ambiguity scoring (≤ 0.2 gate) — validates the spec-first premise
- [ ] Immutable Seed generation (YAML format) — the cryptographic foundation for lineage
- [ ] Cross-model holdout test generation with encryption — the adversarial layer that differentiates from all competitors
- [ ] Bead DAG decomposition with molecules/beads hierarchy — the structural backbone for parallel execution
- [ ] Parallel execution engine with atomic claim and fresh-context-per-bead — prevents context rot and race conditions
- [ ] Testing cube (unit + integration + E2E) generated per feature — proves equal depth
- [ ] At least one full evolutionary cycle (post-execution evaluation → new seed → re-execute) — proves the loop works
- [ ] Holdout unsealing and final validation pass — closes the pipeline
- [ ] Web dashboard: chat interview UI + DAG visualization + streaming logs — makes the pipeline observable
- [ ] CLI interface — alternative surface for pipeline operations
- [ ] Multi-model routing via Vercel AI SDK — enables cross-model holdout diversity

### Add After Validation (v1.x)

- [ ] Brownfield codebase knowledge graph (codebase-memory-mcp) — needed once v1 test case proves greenfield pipeline. Trigger: first non-trivial brownfield project.
- [ ] Git-push triggered pipeline runs — trigger: first team adoption use case
- [ ] Per-project model configuration overrides — trigger: first user wanting different model mix
- [ ] Lateral thinking persona activation UI — trigger: when stagnation detection is proven to work
- [ ] Advanced convergence tuning controls — trigger: after observing real convergence behavior in production

### Future Consideration (v2+)

- [ ] Digital twins (mock servers for third-party integrations) — prerequisite for Cauldron building production apps that integrate with Clerk, Stripe, AWS
- [ ] Deployment pipeline integration — v2, after development pipeline is proven
- [ ] Multi-tenant SaaS hosting — v2+, after OSS release
- [ ] Real-time collaboration (multiple users on same project) — v2+, significant infrastructure complexity
